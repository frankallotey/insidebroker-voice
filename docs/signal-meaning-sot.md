# INSIDE BROKER — Signal Meaning SOT v2.0

STATUS: LOCKED
AUTHORITY: CONSTITUTIONAL FOR SIGNAL INTERPRETATION DOMAIN
SCOPE: BACKEND DOMAIN LAYER, SURFACE CONSUMPTION CONTRACTS, INTERPRETATION ENGINE GOVERNANCE

## CANONICAL ARCHITECTURE

```
Signal → Constraint Layer
       → AI Interpretation
       → Meaning Object
       → Surfaces
```

Constraint layer: deterministic authority
AI layer: bounded contextual refinement only
Surfaces: pure renderers only

This is the non-negotiable architectural flow.
No implementation may deviate from it.

---

## 1. PURPOSE

This document defines the canonical policy layer that interprets raw signals into structured meaning objects.

It establishes:

- what a signal is and is not
- what meaning is and is not
- how meaning is classified
- how meaning is generated
- what each surface may and may not do with meaning
- what is forbidden
- what is deferred
- what is frozen

This is not a UI specification.
This is not an implementation guide.
This is not a product description.

It is a frozen domain policy.
All implementation must conform to it without exception.

## 2. PROBLEM STATEMENT

Inside Broker detects signals from live conversations.

A signal says:

> "This happened."

That is insufficient for three reasons.

### REASON 1 — AMBIGUITY

The same signal type carries different weight depending on:

- conversation stage
- product context
- session state
- signal sequence
- regulatory disclosure timing

`OBJECTION_PRICE` at minute 2 of an opening conversation is not the same event as `OBJECTION_PRICE` at minute 18 of a closing conversation after value has been explained twice.

A flat signal-to-meaning lookup produces the wrong interpretation in at least one of these cases.

### REASON 2 — SURFACE DIVERGENCE

Three downstream surfaces consume signals:

- advisor dashboard
- compliance dashboard
- regulatory evidence mode

Without a shared interpretation layer each surface would interpret signals independently.

This produces drift.

In a regulated environment drift is not a UX problem. It is a governance failure.

### REASON 3 — REGULATORY INDEFENSIBILITY

A raw signal cannot appear in a Consumer Duty evidence pack.

> "OBJECTION_PRICE was detected" is not evidence of anything.

> "A price objection was raised at a stage where value explanation was required. The advisor was guided to address it. The guidance was acknowledged." is evidence.

The meaning layer produces the second form from the first.

## 3. CANONICAL DEFINITION OF SIGNAL

A signal is a structured event emitted by the signal detection layer indicating that a conversation condition of potential significance has been observed.

A signal has:

- a type (what was detected)
- a timestamp (when it occurred)
- a session reference (where)
- a source (how it was detected)

A signal does NOT have:

- interpretation
- severity
- recommended action
- regulatory context
- confidence weighting

A signal is a detection event. Nothing more.

Interpretation is the exclusive responsibility of the meaning layer defined in this document.

### SIGNALS ARE INTERNAL

Signals are internal system events.

Surfaces do not consume signals. Surfaces consume meaning objects.

This is absolute. No surface may access raw signals directly under any circumstances.

---

## 4. CANONICAL DEFINITION OF SIGNAL MEANING

A signal meaning object is the canonical interpretation of a signal within its full interpretation context.

It answers:

- What does this signal imply?
- How serious is it in this context?
- Why does it matter regulatorily?
- What should happen next?
- How confident is the system in this interpretation?

A signal meaning object is:

- generated by the backend domain layer only
- consumed by downstream surfaces unchanged
- never generated in the UI
- never generated independently per surface
- mutable during an active session as context accumulates
- frozen at session end
- immutable after freezing

ONE meaning object is produced per signal.

## 5. MEANING CLASSIFICATION MODEL

Five canonical meaning classes exist.

No others are permitted.
No implementation may introduce additional classes.
No surface may define its own classification logic.

---

### CLASS 1 — ADVISORY

**Definition:**
A guidance opportunity has been detected. No compliance concern exists. The advisor may benefit from intervention.

- Response required: OPTIONAL
- Advisor UI: amber card, dismissible
- Compliance surface: pattern analysis only
- Evidence surface: contextual only
- Severity range: LOW only

---

### CLASS 2 — CAUTION

**Definition:**
A potential compliance issue is developing. No breach has occurred. Advisor awareness is required. The situation may resolve without intervention.

- Response required: ACKNOWLEDGEMENT EXPECTED
- Advisor UI: amber card, acknowledgement expected
- Compliance surface: individual flag, monitoring required
- Evidence surface: included with resolution outcome
- Severity range: LOW to MEDIUM

---

### CLASS 3 — BREACH

**Definition:**
A compliance requirement has been violated or a mandatory obligation has been missed.

This class is owned exclusively by DutyGuard and the constraint layer.

The AI interpretation engine has zero authority over BREACH classification. This is absolute and non-negotiable.

A BREACH may block conversation progression in the advisor surface.

- Response required: MANDATORY IMMEDIATE
- Advisor UI: red DutyGuard card, non-dismissible, may block progression
- Compliance surface: critical flag, review required
- Evidence surface: primary compliance event, full audit trail required
- Severity range: HIGH to CRITICAL

---

### CLASS 4 — INCOMPLETE

**Definition:**
A required element was not completed by session end.

This class is assigned retrospectively only. It cannot exist during an active session. It is created during session close processing exclusively.

- Response required: REVIEW REQUIRED
- Advisor UI: post-call summary only, never during active call
- Compliance surface: follow-up action required
- Evidence surface: primary evidence of unresolved obligation
- Severity range: MEDIUM to HIGH

---

### CLASS 5 — POSITIVE

**Definition:**
Confirmed compliant behaviour or good outcome has been observed.

This class requires dual confirmation:
- signal indicating compliant behaviour observed
- constraint layer confirmation that required element was genuinely completed

The AI engine alone cannot assign POSITIVE class.

POSITIVE is evidential and reinforcement only. It is never real-time guidance.

- Response required: NONE
- Advisor UI: NOT shown during active call. Shown in post-call summary only.
- Compliance surface: governance record, Consumer Duty input
- Evidence surface: primary input to Consumer Duty evidence pack
- Severity: NONE (POSITIVE class carries no severity)

## 6. SIGNAL MEANING SCHEMA

The canonical signal meaning object.

All fields are required unless explicitly marked optional.

```
SignalMeaning {

  -- Identity

  id: UUID

  signal_id: UUID
    Reference to source signal.

  session_id: UUID

  -- Classification
  -- (assigned by constraint layer, never by AI)

  meaning_class: MeaningClass
    ADVISORY | CAUTION | BREACH | INCOMPLETE | POSITIVE

  severity: Severity | null
    LOW | MEDIUM | HIGH | CRITICAL
    Null for POSITIVE only.
    Range is constrained per class by the constraint layer.
    AI may refine within range.
    AI may not exceed range.

  -- Interpretation

  title: string
    Short human-readable label.
    Maximum 80 characters.
    Must be intelligible without additional context.

  explanation: string
    Plain English description of what was detected and why it matters in this context.
    Written for a compliance professional.
    Must reference the interpretation context that influenced the interpretation.
    AI-generated within approved output constraints.

  regulatory_basis: string
    The regulatory principle or Consumer Duty obligation that makes this signal meaningful.
    Human-authored per signal type and meaning class combination.
    Version-controlled.
    NEVER AI-generated.
    Appears in evidence packs.
    Must be stable and auditable.

  recommended_action: string | null
    What should happen next.
    Null for POSITIVE class.
    Null for INCOMPLETE class.
    AI-generated within constraint bounds.

  -- Context Envelope
  -- (split into two layers)

  interpretation_context: {
    conversation_stage: string
    signal_sequence_position: integer
    prior_signals_of_same_type: integer
    disclosure_state: string
    session_state: string
    product_context: string | null
  }
    Conditions that MAY influence classification and severity.
    Snapshot at time of interpretation.
    Immutable once written.

  profiling_context: {
    advisor_risk_tier: string | null
    network_policy_flags: string[] | null
    prior_session_patterns: string[] | null
  }
    Conditions that MAY influence severity and monitoring only.
    MUST NOT influence classification.
    This constraint is absolute.
    Advisor history must never change meaning class.

  -- Confidence and Source

  confidence: ConfidenceLevel
    DETERMINISTIC | HIGH | MEDIUM | LOW

    DETERMINISTIC: assigned by constraint layer without AI.
    Always used for BREACH.
    Always used for DutyGuard triggers.
    Always used for POSITIVE.

    HIGH / MEDIUM / LOW: AI interpretation within constraint bounds.

  interpretation_source: InterpretationSource
    CONSTRAINT_LAYER | AI_ENGINE | HYBRID

    CONSTRAINT_LAYER: fully deterministic, no AI.
    AI_ENGINE: AI-assigned within constraint bounds.
    HYBRID: constraint layer set class and severity bounds, AI refined within those bounds.

  -- Evidence

  evidence_linkage: {
    included_in_evidence: boolean
    evidence_weight: PRIMARY | SUPPORTING | CONTEXTUAL | null
  }

  -- Human Validation
  -- (mandatory fields, populated post-session)

  human_validation_status: ValidationStatus
    PENDING | VALIDATED | CHALLENGED | OVERRIDDEN

    PENDING: not yet reviewed.
    VALIDATED: compliance officer confirmed system interpretation is correct.
    CHALLENGED: flagged for further review. Does not alter meaning object. Audit trail entry created.
    OVERRIDDEN: compliance officer has replaced classification. Original classification preserved in audit trail.
      Override is explicit, never silent, always auditable.
      Override requires:
        - validated_by
        - validation_timestamp
        - override_reason (mandatory)

  validation_outcome: string | null
    Human-authored note on validation decision.
    Required if status is OVERRIDDEN.

  validated_by: UUID | null
    User ID of compliance officer who performed validation.

  validation_timestamp: timestamp | null

  -- Lifecycle

  created_at: timestamp

  revised_at: timestamp | null
    Last revision during active session. Null if never revised.

  frozen_at: timestamp | null
    Populated at session end. Null during active session.

  frozen: boolean
    false during active session.
    true after session end.
    Frozen objects are immutable.
    No field may be modified after frozen = true.
    Exception: human validation fields may be populated after freeze as part of compliance review workflow.

}
```

## 7. RUNTIME INTERPRETATION RULES

### RULE 1 — CONSTRAINT LAYER RUNS FIRST WITHOUT EXCEPTION

Before the AI interpretation engine is invoked the constraint layer evaluates every signal.

If the constraint layer produces a deterministic meaning assignment the AI engine is not invoked for classification.

The constraint layer owns:
- all BREACH classifications
- all DutyGuard-triggered events
- all POSITIVE confirmations
- the meaning_class assignment for every signal
- the severity range permitted for every class and signal combination

### RULE 2 — AI ENGINE SCOPE IS STRICTLY LIMITED

The AI interpretation engine may only:

- generate explanation text
- generate recommended_action text within approved output constraints
- refine severity within the range set by the constraint layer
- assign confidence level
- assess contextual weight within interpretation_context

The AI engine may NEVER:

- assign meaning_class
- override a constraint layer classification
- override a DutyGuard finding
- exceed the severity range set by the constraint layer
- produce a meaning class not defined in this SOT
- produce an unexplainable output
- use profiling_context to influence classification
- generate regulatory_basis content

Every AI output must be accompanied by a human-readable explanation of why that interpretation was produced. "The model decided" is never an acceptable explanation.

### RULE 3 — CLASSIFICATION IS DETERMINISTIC

All meaning_class assignments are deterministic.

They are produced by the constraint layer based on explicit rules.

Probabilistic classification is forbidden in v1.

### RULE 4 — PROFILING CONTEXT CANNOT INFLUENCE CLASSIFICATION

Advisor history, behavioural patterns, and network policy flags exist in profiling_context.

profiling_context may influence:
- severity within permitted range
- monitoring escalation
- compliance review prioritisation

profiling_context may NEVER influence:
- meaning_class assignment
- regulatory_basis content
- evidence_weight

This constraint is absolute.

### RULE 5 — MEANING IS MUTABLE DURING SESSION

A meaning object may be revised during an active session as context accumulates.

Revisions are logged with revised_at timestamp. Full revision history is preserved in the audit trail.

Example: CAUTION assigned at minute 5 may be revised to POSITIVE at minute 12 if the advisor correctly addresses the concern.

### RULE 6 — MEANING IS FROZEN AT SESSION END

When a session ends all meaning objects are frozen.

```
frozen = true
frozen_at = session end timestamp
```

Frozen meaning objects are immutable. No revision is possible after freeze. This is absolute.

### RULE 7 — INCOMPLETE CLASS IS RETROSPECTIVE ONLY

INCOMPLETE meaning objects are created during session close processing exclusively. They do not exist during an active session.

### RULE 8 — POSITIVE CLASS REQUIRES DUAL CONFIRMATION

POSITIVE meaning objects require:
- a signal indicating compliant behaviour observed
- constraint layer confirmation that the required element was genuinely completed

AI engine alone cannot assign POSITIVE class.

### RULE 9 — REGULATORY_BASIS IS HUMAN-AUTHORED AND VERSIONED

regulatory_basis is drawn from a versioned human-authored lookup keyed by signal type and meaning class combination.

It may not be generated, paraphrased, or inferred by the AI engine.

Changes to regulatory_basis content are version-controlled and auditable.

## 8. SURFACE CONSUMPTION CONTRACTS

These are locked contracts. They define exactly what each surface may and may not do. No implementation may deviate from them.

---

### ADVISOR DASHBOARD CONTRACT

**CONSUMES:**
- ADVISORY meaning objects
- CAUTION meaning objects
- BREACH meaning objects

**DOES NOT CONSUME:**
- POSITIVE meaning objects during active call
- INCOMPLETE meaning objects during active call
- Raw signals under any circumstances

**POST-CALL SUMMARY CONSUMES:**
All meaning classes including POSITIVE and INCOMPLETE

**RENDERING RULES (LOCKED):**

BREACH meaning object:
- → Red DutyGuard card
- → Non-dismissible
- → DutyGuard authority label visible
- → May block conversation progression
- → Acknowledgement mandatory before proceeding

CAUTION meaning object:
- → Amber guidance card
- → Acknowledgement expected
- → Dismissible after acknowledgement only

ADVISORY meaning object:
- → Amber guidance card
- → Optional
- → Dismissible without acknowledgement

INCOMPLETE meaning object:
- → Post-call summary only
- → Never during active call

POSITIVE meaning object:
- → Post-call summary only
- → Never during active call

**SIGNAL PANEL CONTRACT (LOCKED):**

The SignalPanel is input-only.

Pressing a signal button:
- → triggers backend signal event
- → backend produces meaning object
- → meaning object returned to UI
- → UI renders meaning object

The UI must NOT:
- → display signal acknowledgement
- → interpret the signal directly
- → produce any output from a signal before receiving a meaning object from backend

**FRONTEND CLASSIFICATION RULE:**

The advisor dashboard must not implement any classification logic.

It must render meaning_class exactly as received from the backend meaning layer.

No exceptions.

---

### COMPLIANCE DASHBOARD CONTRACT

**CONSUMES:** All meaning classes

**RENDERING RULES:**

ADVISORY:
- → Pattern analysis aggregation
- → Not individually flagged
- → Visible in session overview

CAUTION:
- → Individual flag
- → Monitoring required indicator
- → Resolution outcome visible

BREACH:
- → Critical flag
- → Review required indicator
- → Full meaning object visible including context envelope, regulatory_basis, confidence, interpretation_source

INCOMPLETE:
- → Follow-up action required
- → Shown per session

POSITIVE:
- → Governance record
- → Consumer Duty input
- → Visible in evidence summary

**HUMAN VALIDATION CONTRACT:**

The compliance dashboard surfaces the human validation workflow.

Compliance officers may:

VALIDATE:
Confirms system interpretation.
`human_validation_status → VALIDATED`
Audit trail entry created.

CHALLENGE:
Flags for further review. Does not alter meaning object.
`human_validation_status → CHALLENGED`
Audit trail entry created.

OVERRIDE:
Replaces classification with compliance officer judgement.
`human_validation_status → OVERRIDDEN`

RULES FOR OVERRIDE:
- override_reason is mandatory
- original classification preserved in audit trail
- override is never silent
- validated_by populated
- validation_timestamp populated
- override is fully auditable

Overrides feed the training corpus as labelled correction data.

---

### REGULATORY EVIDENCE MODE CONTRACT

**CONSUMES:** Frozen meaning objects only.

No meaning objects are generated at evidence pack creation time. Evidence is assembled from existing frozen objects only.

**EVIDENCE WEIGHT RULES:**

```
BREACH      → PRIMARY evidence
INCOMPLETE  → PRIMARY evidence
POSITIVE    → PRIMARY evidence
  (Consumer Duty requires positive outcome evidence not only breach records)
CAUTION resolved   → SUPPORTING
CAUTION unresolved → PRIMARY
ADVISORY    → CONTEXTUAL only
```

**TRACE REQUIREMENT:**

Evidence mode must include the full trace per meaning object:

- source signal
- interpretation context
- constraint layer decision
- AI refinement if applicable
- confidence level
- interpretation source
- regulatory_basis
- human validation status and outcome if reviewed

This trace is the regulatory defence record.

---

### CONTROL ROOM CONTRACT

**OBSERVES:**
- Meaning layer system health
- Meaning generation metrics
- Constraint layer behaviour
- AI engine behaviour

**DOES NOT CONSUME:**
Individual meaning objects for display to end users

**MUST NEVER:**
- Generate meaning
- Modify meaning
- Reinterpret meaning
- Display individual session meaning objects to operators

**SYSTEM HEALTH METRICS THE CONTROL ROOM MAY SURFACE:**

- Meaning generation rate
- BREACH frequency across network
- AI engine confidence distribution
- Constraint layer override frequency
- Validation coverage rate (proportion of meaning objects human-validated)
- Meaning layer latency
- AI engine error rate

**ALERTS FEED:**

Meaning layer failures surface as alerts:
- AI engine error or timeout
- Output outside constraint envelope detected
- Constraint layer rule evaluation failure
- Unusually high LOW confidence rate (potential signal quality issue)

**FIX QUEUE:**

Meaning layer failures produce Fix Queue items. Human-in-the-loop investigation required. AI-assisted diagnosis permitted within IB-controlled boundaries.

**SYSTEM MAP:**

The meaning interpretation layer must appear as a distinct node in the System Map sitting between the Signal Channel and downstream surfaces. Its status must be independently observable and diagnosable.

## 10. INTERPRETATION ENGINE EVOLUTION

### V1 IMPLEMENTATION

The interpretation engine in v1 uses an external AI provider behind an abstraction interface.

Provider: Anthropic Claude API (same provider as Diagnostic Agent — already in stack)

All calls to the external provider must enforce zero data retention.

Zero data retention means:
- inputs are not stored by the provider
- inputs are not used to train provider models
- this must be explicitly configured on every API call to the interpretation engine

This is non-negotiable. No conversation data may be used to train external models.

### ABSTRACTION INTERFACE (LOCKED)

The AI provider must sit behind a fixed abstraction interface:

```
InterpretationEngine
  → InterpretationInterface
  → Provider (Anthropic v1)
```

This interface must be designed for provider replacement without changes to any downstream system.

Same pattern as EmailService and EncryptionService abstractions already in the codebase.

### TRAINING CORPUS (PROPRIETARY)

Every meaning assignment must be logged internally with:

- input signal
- interpretation context
- constraint layer decision
- AI engine output
- final meaning object
- human validation outcome when available

This log is Inside Broker proprietary IP.

It is never shared with any external provider.
It is never used as input to external model training.
It is stored internally and access-controlled.

This corpus is the foundation of the proprietary interpretation model.

### PROPRIETARY MODEL PATH

A proprietary interpretation model is a planned future replacement for the external AI provider.

Parallel repo: insidebroker-intelligence

Scope:
- custom interpretation model trained on IB corpus
- fine-tuned on UK insurance advice conversations
- calibrated against compliance officer validation outcomes
- no downstream changes required when swapped in

Timeline:
- parallel track only
- funded by pilot traction
- not a prerequisite to pilot
- plugs in behind abstraction interface when ready

### DATA PRIVACY CONSTRAINT (NON-NEGOTIABLE)

No conversation data is used to train external models.

Zero data retention must be enabled on all AI interpretation calls.

All learning remains internal to Inside Broker.

This constraint is absolute and may not be waived.

## 11. EXPLICIT NON-GOALS

This SOT does not define:

1. How signals are detected. Signal detection is defined in the Signal Channel SOT.

2. How meaning objects are displayed visually in any UI. Display rules belong to each surface's own SOT.

3. How the AI engine is implemented technically. This SOT defines its constraints and output contract only.

4. Per-tenant customisation of meaning classes. Meaning classes are canonical and universal. Tenants may configure constraint layer policy within permitted bounds but may not add or remove meaning classes.

5. How evidence packs are formatted or delivered. Evidence formatting belongs to Regulatory Evidence Mode SOT.

6. Advisor scoring or ranking based on meaning objects. Scoring is explicitly forbidden as a direct output of this layer.

7. Cross-session pattern detection. Meaning operates at session level only in v1.

---

## 12. DEFERRED ITEMS

1. Per-tenant constraint configuration UI. Tenants will eventually configure constraint layer floors and ceilings for their network policy. Deferred to future Installer extension.

2. Cross-session pattern analysis. Advisor consistently producing INCOMPLETE objects is a future analytics layer.

3. Meaning versioning and migration strategy. If classification rules change, existing frozen meaning objects must not be retroactively reclassified. Strategy deferred until first classification change is required.

4. Customer-facing meaning objects. Communication derived from meaning objects for customer delivery is a future product layer.

5. insidebroker-intelligence implementation. Proprietary model build is deferred and parallel. Not blocking pilot.

6. Context Enrichment integration. CRM-derived context in the interpretation context envelope is deferred pending Context Enrichment implementation (post v1 Installer).

## 13. FROZEN ARCHITECTURAL PRINCIPLES

**PRINCIPLE 1**
Meaning is generated in the backend domain layer only. Never in the UI. Never in a frontend component. Never in a display layer.

**PRINCIPLE 2**
Meaning is canonical and shared. One meaning object per signal. Consumed unchanged by all downstream surfaces. Never interpreted differently per surface.

**PRINCIPLE 3**
BREACH class is owned by DutyGuard and the constraint layer exclusively. The AI engine has zero authority over BREACH classification. This is absolute.

**PRINCIPLE 4**
Classification is deterministic. The constraint layer assigns meaning_class. The AI engine refines within bounds. The AI engine never assigns class.

**PRINCIPLE 5**
Profiling context cannot influence classification. Advisor history may influence severity and monitoring only. Never classification.

**PRINCIPLE 6**
Every meaning assignment must be explainable. The explanation field is mandatory. "The model decided" is never an acceptable explanation.

**PRINCIPLE 7**
Frozen meaning objects are immutable. No modification after session end. Exception: human validation fields may be populated after freeze as part of compliance review workflow. No retroactive reclassification.

**PRINCIPLE 8**
regulatory_basis is human-authored and version-controlled. It is never AI-generated. It is the regulatory anchor of the entire evidence layer.

**PRINCIPLE 9**
The constraint layer runs before the AI engine on every signal. Without exception.

**PRINCIPLE 10**
Zero data retention is enforced on all AI interpretation calls. No conversation data is used to train external models. All learning is internal. This may not be waived.

**PRINCIPLE 11**
Signals are internal. Surfaces consume meaning objects. No surface may access raw signals directly.

**PRINCIPLE 12**
Frontend surfaces are pure renderers. All interpretation logic lives in the backend domain layer. Frontend must not contain classification logic, interpretation logic, or severity assignment logic.

**PRINCIPLE 13**
The Control Room observes the meaning layer as a system. It does not consume individual meaning objects for user display. It does not generate meaning. It does not modify meaning. It does not reinterpret meaning.

**PRINCIPLE 14**
Human validation of meaning objects is part of the system design not an optional feature. Validation outcomes feed the training corpus. Overrides are explicit, auditable, and never silent.

**PRINCIPLE 15**
This layer has no UI. It is infrastructure. Its outputs are consumed by surfaces that have UI. The meaning layer itself is invisible to end users and that is correct.
